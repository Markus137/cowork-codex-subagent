# GitHub quality-gate rationale

The productive and review workflow is GitHub-only. It does not require the
user to clone a repository or operate a local Git worktree.

## Official OpenAI path

- [Codex cloud](https://developers.openai.com/codex/cloud) connects GitHub,
  runs tasks in isolated cloud environments, and supports starting and
  reviewing work without tying it to the user's development machine.
- [Codex code review in GitHub](https://developers.openai.com/codex/cloud/code-review)
  is the primary quality gate. OpenAI documents repository setup at Codex Code
  review settings, Automatic reviews, the exact `@codex review` trigger,
  current GitHub review comments, `AGENTS.md` review guidance, and the fact that
  GitHub reviews flag P0/P1 issues only.
- [The official Codex repository](https://github.com/openai/codex) is the
  upstream implementation and issue tracker.
- [The official Claude/Codex plugin](https://github.com/openai/codex-plugin-cc)
  remains useful in Claude Code terminals, but its Bash-based rescue subagent
  is not the Cowork transport. This plugin's host MCP owns the one-way initial
  job and single possible Codex CLI `resume` transport.

For the subscription path, setup uses the existing ChatGPT/Codex login plus
the connected GitHub account. This workflow never requests an OpenAI API key,
adds a GitHub Action, or invokes a local review runner.

The Codex runtime's automatic approval reviewer is a separate permission
boundary for side-effecting GitHub MCP calls. The worker combines
`approval_policy="on-request"`, `approvals_reviewer="auto_review"`, GitHub
write approval mode, strict configuration parsing, the built-in guardian
baseline, and an additional run-specific restriction. It never uses yolo or
full-access mode. This reviewer can approve or deny a tool request, but it does
not research, implement, review code quality, act as Fable/SOL/Terra, or count
as an iteration. Managed requirements remain authoritative and failures are
reported explicitly rather than bypassed.

## Bounded review circuit

The ready pull request receives one initial Automatic review. Its create result
and the final host-observed `github.get_pr_info` read must both return the PR
body with exactly one original run/created-head PENDING marker. The canonical
no-period line must be derived from the latest successful task-branch commit
already known before PR creation. New PR creation accepts only that exact line
with no other bytes. The host retains read compatibility for a persisted legacy
certification containing the identical line with one terminal ASCII period and
requires the same observed variant across later final reads. Duplicate, mixed,
decorated, padded, or otherwise punctuated variants fail. The host also
explicitly allowlists the known `github.fetch_pr` runtime equivalent, but no
other similarly named tool. A correction may change the PR head while that
historical marker remains bound to `prCertification.createdHeadSha`. Two narrow MCP
validators enforce evidence binding without network or repository access.
They accept only the configured bot login (case-fold plus optional final
`[bot]` normalization), strict post-boundary timestamps, an unchanged full
head with zero intervening head-change events, and exact GitHub targets.

Static source inspection confirms that the marker-only create-body contract
also applies to implementation PRs. The bounded `Problem`, `Change`,
`Rationale`, and scope-bearing `Verification` explanation therefore lives
inside the already necessary implementation commit message under the
run-bound `COWORK_CODEX_IMPLEMENTATION_V1` marker. The host validates it before
the commit mutation, binds the returned SHA to the exact task branch, and
requires the final PR head to equal the last such branch-effective commit.
There is no extra context-comment write or comment-fetch proof. The commit
explanation is provenance and is never quality-gate evidence.

Initial clean may be the bot's later `+1` reaction directly on the recorded PR,
but only authoritative GitHub Issues/Reaction metadata may establish its
mechanical actor login, full timestamp, exact target type/ID, and canonical
URL. Browser DOM, minute-only or relative labels, counters, and inference do
not count. Missing `Issues: Read` or equivalent authoritative reaction data is
blocked/incomplete and can never produce PASS; the plugin cannot grant that
permission. Initial `issue_comment` events are rejected.

The other initial evidence family is a real GitHub review bundle: the bot
review submission body carries one plain or Markdown 10–40 hex
`Reviewed commit:` prefix, while zero or more separate bot inline comments
link back through the same `pull_request_review_id`. The prefix must match the
recorded full head and exactly one known PR commit. With `comments: []`, the
bundle is clean only if the submission body has true absence of priority-like
material. With comments, each genuine inline body supplies exactly one
canonical P0–P3 badge/list marker, and the validator preserves its exact body,
URL, path, and line for forwarding; those four fields are the immutable body, URL, path, and line. The scanner distinguishes absence,
canonical, malformed, unknown P4+, and multiple material. Stale, short,
nonhex, wrong, unassociated, mixed, malformed, duplicate, or ambiguous
evidence never counts. Inline comments may precede submission time, but both
must be strictly later than the stage boundary.

If the initial review finds P0–P3 issues, Fable forwards their immutable text
byte-for-byte once to the same SOL job. SOL and Terra may update the same
remote branch once. Fable then posts one standalone comment whose entire body
is `@codex review`. Final clean accepts the bot's later authoritative `+1`
reaction on that exact new issue-comment ID and canonical URL for backward
compatibility, or a valid zero-comment review bundle. A PR-level thumbs-up is
invalid and temporary `eyes` reactions are ignored. Final findings remain raw
review-submission plus associated inline-comment evidence with the same exact
actor/time/full-head/unique-prefix binding.

A later final bot `issue_comment` can be clean only when its positive unique
ID differs from the trigger ID, its canonical issue-comment URL and exact PR
number/URL match, its authoritative actor/time is strictly after the trigger,
the head never changed, its body has exactly one matching Reviewed-commit
marker and true absence of priority-like material, and `review_comments` is
explicitly `[]`. “Didn't find...” prose is never evidence. Finding-like issue
comments are rejected because they have no authoritative raw inline
URL/path/line mapping; Fable may not synthesize one. Malformed, unknown P4+,
duplicate, ambiguous, or multiple priority/commit material fails closed.

The worker-created PR body retains its exact run/created-head-bound
`PENDING / DO NOT MERGE` marker. Only after the final validator accepts clean
evidence and Fable reconfirms the unchanged final head does Fable post exactly
one connector comment whose full body is
`COWORK_CODEX_GATE_V1 | run_id=<RUN_ID> | head_sha=<FINAL_40_HEX> | PASS`.
That later same-run/same-head comment is the certification; it does not rewrite
the historical PENDING body. A post failure is incomplete, and blocked,
findings, timeout, stale, ambiguous, or changed-head outcomes receive no PASS.
Codex/SOL/Terra never post PASS, and nobody edits the PR body, toggles draft,
closes, merges, or deletes through this gate. The workflow then stops regardless of outcome. There is no initial manual trigger, second
correction, repeated final trigger, third review, Stop-hook loop, or retry-
until-clean behavior.

No automatic PR retry is part of the gate. If an audit artifact commit exists
but PR creation fails, the host publishes an
`audit_artifact_committed_pr_missing` residue bound to the exact unchanged
repository, base, task branch, head, and artifact. Manual recovery may open one
PR with the required canonical marker; it must not mutate, merge, delete, close,
or clean up. Active approvals leave the host job `running`; terminal
`approval_pending` model envelopes are invalid because the stream has no stable
separate in-flight approval signal.

Running progress is event-bound, not timer-bound. A successful exact-repository
GitHub event can publish sanitized `branch_created`, `commit_observed`,
`pr_created`, or `pr_verified` state; unchanged read-only activity is throttled
to one `updated_at` refresh per minute. The fixed structured wall-clock cap is
45 minutes by default (allowed 15–120) and is never reset by progress. There
is no separate enforced or default 30-minute SOL manager budget.
Alleged approval wait cannot be excluded because no stable pending signal
exists. Later paired runs showed Bug 8 as an inconsistent approval-policy
interpretation: both Terra-owned PR calls contained extra body text, but one
was denied and the other allowed. The workflow neither retries nor bypasses
that decision; Terra is read-only, SOL owns the main-thread mutation, and new
PR creation uses exactly the canonical marker with no other bytes. Bug 10 was fixed in v1.2.3 through
same-repository event-bound progress and `updated_at`; Bug 11 was mitigated in
v1.2.3 through the one-shot structured 45-minute default with explicit 15–120
bounds. The measured delay model is reasoning/timing evidence, not approval.

Version 1.2.9 replaces the extra implementation context-comment write with a
bounded, run-bound explanation in the already necessary final branch-effective
commit. Direct and low-level commit paths, no-op resume, real corrections, and
the final byte-exact marker-only PR body are host-validated. Version 1.2.8 added one bounded, same-PR, host-verified implementation context
comment while retaining the exact marker-only PR body. The write has one
attempt, terminal write failures preserve the branch and PR, audit creates no
comment, and resume cannot duplicate verified proof. Version 1.2.7 adds the terminal-fixture matrix, makes SOL own every observable
GitHub mutation and final proof read, and derives future audit evidence from
the verified committed report. Version 1.2.6 added the single-source generated
18-field result-envelope contract and sanitized expected type/shape in validation errors. Current
hardening makes the report machine block the sole future audit-evidence source:
the model returns `audit_evidence:null`, the host derives and validates the
evidence from the exact-head full report and exact-range reads, and only then
hydrates the public result. A legacy non-null model value must match that
derived evidence exactly. Version 1.2.5
added a pre-PR audit gate: the last completed report write must
contain one parseable, strict, contract-bound evidence block and must match a
full self-fetch plus every same-SHA ranged fetch before PR creation can start.
Parse, schema, binding, and later artifact/envelope mismatches remain separate
fail-closed outcomes. Terminal status unions fresh observations with persisted
residues. This fixes Bugs 16–18 and keeps Bug 6 as an explicit reachable
`AUDIT_EVIDENCE_UNVERIFIED` path. Bugs 8, 10, 11, and 14 retain their prior
disposition unless directly affected by this pre-PR gate.

Terra is read-only: it inspects, analyzes, and drafts V1/V2. After accepting
V2, SOL alone performs every GitHub mutation and final proof read in the main
thread, making create, range, report, and final PR observations visible to the
host collector. A new PR body contains only the canonical PENDING marker and
no summary or blank line. The historical terminal-period form remains
read-compatible but is never emitted. If a PR write is denied after a commit,
an exact zero-delta comparison binds a candidate full SHA to the task branch;
an audit-specific residue additionally requires a successful full fetch of the
exact audit path at that SHA.

Codex authors the report and evidence block via GitHub MCP. The host never
writes, inserts, replaces, or repairs report content. Its
`canonicalAuditBlock()`/`auditEvidenceBlock()` helpers serialize only the
expected comparison object used by the validator and deterministic tests; the
gate parses the model-authored report write before PR creation.

Snippet comparison is exact and performs no whitespace, final-newline,
Unicode, or CR/LF normalization. Pre-PR and final validation use one shared
strict audit-evidence normalizer, with the legacy verification-array exception
limited to final-envelope input compatibility. Terminal merge emits at most
two resources and always lets a trusted PR supersede a stale commit-without-PR
residue.

## What is borrowed from Steipete

[Steipete's current `codex-review` skill](https://github.com/steipete/agent-scripts/blob/main/skills/codex-review/SKILL.md)
treats findings as advisory, verifies them against the actual code, reruns
focused tests after fixes, and re-reviews changed code. We retain that useful
separation: Fable transports findings unchanged, while SOL verifies them and
Terra applies any justified correction.

We intentionally do not copy Steipete's default yolo/full-access helper mode
or an unbounded "until clean" loop. GitHub remains the shared mailbox and the
second review is a hard circuit breaker.
